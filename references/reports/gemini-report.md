Переход от программной реализации RAG на Python и LangChain к low-code оркестратору n8n требует изменения подхода к управлению потоками данных. n8n оперирует JSON-объектами (Items), передаваемыми между узлами, где каждый узел инкапсулирует логику, которую в коде вы бы писали вручную (например, сплиттеры, эмбеддеры, коннекторы к БД).

Ниже представлен план реализации MVP для компании «Авиастар», архитектура решения и схема бизнес-процесса, адаптированные под стандарты технической документации.

### 1. Схема бизнес-процесса (BPMN Swimlanes)

Для описания потока используется стандарт Mermaid. Эту схему можно интегрировать в вашу документацию (Docs-as-Code) или отрендерить в визуальный формат для демонстрации ЗГД.

```mermaid
bpmn
    title n8n RAG AI-Assistant (MVP)

    lane Сотрудник (Telegram)
        node "Отправка запроса" as start
        node "Получение ответа" as end
    endlane

    lane n8n Orchestrator
        node "Telegram Trigger (Webhook)" as trigger
        node "Проверка Auth (Whitelist)" as auth
        node "Классификация (Q&A / Шаблон)" as router
        node "Генерация по шаблону (Код)" as template_gen
        node "Оценка уверенности (Scoring)" as scoring
        node "Логирование (Google Sheets)" as logging
    endlane

    lane Vector DB (Supabase)
        node "Поиск релевантных чанков" as vector_search
    endlane

    lane LLM Provider
        node "Генерация ответа (LLM Chain)" as llm_gen
    endlane

    start --> trigger
    trigger --> auth
    auth -- "Не в списке" --> end
    auth -- "Успешно" --> router
    router -- "Шаблон" --> template_gen
    router -- "Вопрос" --> vector_search
    vector_search --> llm_gen
    llm_gen --> scoring
    scoring -- "Низкий скор" --> end
    scoring -- "Высокий скор" --> logging
    template_gen --> logging
    logging --> end
```

### 2. Стратегия развертывания: Docker (Local/WSL) -> HuggingFace Spaces

Развертывание n8n в HuggingFace Spaces требует подготовки кастомного `Dockerfile`, так как стандартный образ n8n нуждается в адаптации прав доступа для работы в среде HF (где контейнеры запускаются от непривилегированного пользователя, но требуют записи в определенные директории).

**Ключевые шаги для HF Spaces:**
1.  Создайте пространство (Space) с типом Docker.
2.  Настройте `Dockerfile` на базе `n8nio/n8n`.
3.  Определите переменные окружения (Secrets) в настройках Space: `WEBHOOK_URL` (URL вашего Space), токены Telegram, ключи OpenAI/Anthropic, доступы к Supabase.
4.  В `Dockerfile` обеспечьте права на чтение/запись для директории `/home/node/.n8n`.

### 3. Архитектура MVP RAG в n8n

Вместо написания связок на Python, в n8n выстраивается визуальный пайплайн.

*   **Индексация (Отдельный Workflow):**
    *   **Узлы:** Read PDF/Word -> Token Text Splitter (500/50, как в ТЗ) -> Embeddings (OpenAI/Cohere) -> Supabase Vector Store (Upsert).
    *   **Данные:** Используйте открытые демо-данные (например, ТК РФ, ПДД для грузоперевозок).
*   **Чат-бот и Поиск (Основной Workflow):**
    *   **Telegram Trigger:** Слушает входящие сообщения.
    *   **Code Node (Auth):** Простой JS/Python скрипт, проверяющий `chat.id` отправителя по заранее заданному массиву (белому списку). Если ID нет — процесс прерывается (Return empty item).
    *   **AI Agent / LLM Chain Node:** В n8n есть нативные Advanced AI узлы. Используйте узел `Question and Answer Chain`. К нему подключаются:
        *   Модель (LLM).
        *   Узел Vector Store (Supabase) в режиме `Retrieve`.
    *   **Заполнение шаблонов (Строго без LLM):** Используйте узел `Switch` (или `Router` агента) для классификации намерения. Если это "Договор", направляйте поток в классический узел (не AI) для подстановки переменных в текстовый/Word шаблон, чтобы исключить галлюцинации, как требует заказчик.
    *   **Логирование:** Параллельная ветка после отправки сообщения в Telegram, которая записывает `chat.id`, `question`, `answer` и `timestamp` в Supabase или Google Sheets.

### 4. Ответы на требования заказчика (для презентации)

Для успешной защиты кейса перед ЗГД, необходимо предоставить аналитику рисков и встречные вопросы, демонстрирующие уровень понимания корпоративных систем.

**Риски (Опишите риски):**
1.  **Деградация качества поиска при росте базы (Vector Search Limitations):** При увеличении количества документов (свыше тысяч чанков) простой семантический поиск начнет выдавать нерелевантные куски. Потребуется внедрение Hybrid Search (Keyword + Vector) и метаданных (фильтрация по отделам перед векторным поиском).
2.  **Галлюцинации LLM на противоречивых данных:** В нормативной базе часто встречаются устаревшие приказы, противоречащие новым. Без жесткого версионирования и фильтрации по дате метаданных в Supabase, LLM может синтезировать ответ из двух взаимоисключающих регламентов.
3.  **Ограничения n8n в HF Spaces:** Бесплатные инстансы HuggingFace засыпают (scale to zero) при неактивности. Холодный старт займет время, что вызовет таймауты в Telegram при первом запросе. Для продакшена потребуется выделенный сервер.

**Дополнительные Требования и Вопросы (Что нужно от заказчика):**
1.  **Топология прав доступа:** Существуют ли документы, доступ к которым должен быть ограничен внутри самого белого списка (например, HR видит одно, юрист — другое)? Это критично для проектирования схемы метаданных в pgvector.
2.  **Процесс инвалидации знаний:** Кто и как будет инициировать удаление устаревших векторных представлений из БД при отмене внутреннего регламента? Нужен ли интерфейс (или отдельный бот) для администратора базы знаний?

### 2. Поддержка Telegram в n8n
В n8n есть нативная и очень мощная поддержка Telegram[cite: 3]. Вам не нужно писать код для обработки API.

*   **Telegram Trigger:** Работает через Webhook. Он автоматически получает все сообщения, отправленные боту[cite: 3].
*   **Авторизация (Whitelist):** В настройках узла `Telegram Trigger` есть параметры `Restrict to Chat IDs` или `Restrict to User IDs`[cite: 3]. Вы можете просто вписать туда ID разрешенных пользователей через запятую. Это полностью закрывает требование ТЗ по безопасности без написания кода[cite: 3].
*   **Telegram Node:** Используется для отправки ответов, документов или кнопок (например, для оценки ответа «👍/👎»)[cite: 3].

### 3. Мышь против ИИ (Claude/Cursor)
Как опытному программисту, вам может захотеться генерировать весь workflow кодом. 

**Мой совет: используйте гибридный подход.**
*   **Почему не стоит генерировать весь JSON через Claude:** n8n очень чувствителен к версиям узлов и их координатам на холсте. Claude может выдать JSON для версии 1.0, когда у вас стоит 2.1, и импорт просто не сработает или сломает связи.
*   **Идеальный сценарий:**
    1.  Разверните n8n и создайте каркас мышью (это займет 5 минут: перетащить 5-6 узлов и соединить их).
    2.  Используйте **Claude/Cursor** для написания кода **внутри** узла `Code Node` (JavaScript/TypeScript). Например, для сложной очистки DITA XML или специфической трансформации метаданных перед отправкой в Supabase[cite: 3].
    3.  Вы можете копировать отдельные узлы (Ctrl+C в UI n8n копирует JSON узла в буфер), отдавать их Claude на рефакторинг и вставлять обратно (Ctrl+V).

### 4. Диаграмма для swimlanes.io
ТЗ требует схему в "swim lanes". В нотации этого сервиса (Sequence Diagram) это будет выглядеть так:

```text
title: AI-Ассистент HR/ЮрО (MVP)

group Юрист / HR (Telegram)
  User -> TG_Bot: Запрос (вопрос или шаблон)
end

group n8n Orchestrator
  TG_Bot -> Auth: Webhook (Проверка ID в Whitelist)
  Auth -> Router: Классификация (Q&A или Шаблон)
  
  note over Router: Если Q&A:
  Router -> Supabase_Vector: Поиск контекста (pgvector)
  Supabase_Vector --> LLM_Agent: Релевантные чанки
  LLM_Agent -> LLM_Agent: Генерация ответа с источниками
  
  note over Router: Если Шаблон:
  Router -> Code_Node: Заполнение полей (без LLM)
  
  LLM_Agent -> Logger: Запись (User, Query, Score)
  Code_Node -> Logger: Запись события
end

group Хранилище
  Logger -> Google_Sheets: Лог транзакции
  Logger -> Supabase_DB: История запросов
end

n8n Orchestrator -> User: Ответ + кнопки оценки (TG)
```

### MVP: План действий (Первые 24 часа)
1.  **Инфраструктура:** Поднимите n8n в Docker/WSL. Подключите его к Supabase (создайте таблицу через SQL-скрипт, который есть в вашем JSON-шаблоне в узле `Sticky Note`)[cite: 2].
2.  **Индексация:** Используйте верхнюю часть вашего JSON (Top Flow), чтобы загрузить 3-5 тестовых PDF-файлов (регламенты, шаблоны) в Supabase[cite: 2].
3.  **Telegram:** Создайте бота в BotFather, подключите его к `Telegram Trigger` и добавьте свой ID в Whitelist[cite: 3].
4.  **Логика:** Соедините Trigger -> Vector Store -> LLM -> Telegram Node.
