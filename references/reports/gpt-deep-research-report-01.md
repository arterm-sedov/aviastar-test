# Развёртывание n8n (Docker, Hugging Face Spaces)  
Для работы с n8n можно использовать Docker Compose. Официальная документация описывает установку Docker и запуск n8n через docker-compose【46†L24-L33】【28†L725-L733】. Например, в n8n AI Starter Kit есть готовый `docker-compose.yml`, где сервис n8n подключается к базе PostgreSQL (или Supabase) через переменные окружения (DB_TYPE, DB_HOST, DB_USER, DB_PASSWORD и т.д.)【28†L578-L587】【28†L725-L733】. В локальной среде на WSL можно создать `.env` с этими переменными, затем запустить `docker compose up -d`.  

При деплое на Hugging Face Spaces используют Docker SDK: создают своё Space, дублируют шаблон (например, проект Tomo Wang) и задают переменные окружения через интерфейс Spaces【43†L129-L137】. Важный момент – Spaces «усыпают» после простоя, поэтому для хранения данных рекомендуется внешний БД, например Supabase【43†L80-L82】. В переменных Space задают параметры подключения к Supabase (`DB_POSTGRESDB_HOST`, `DB_POSTGRESDB_USER`, `DB_POSTGRESDB_PASSWORD`), а также `N8N_ENCRYPTION_KEY`, `WEBHOOK_URL` и `N8N_EDITOR_BASE_URL` (обычно вида `https://<имя>-n8n.hf.space/`)【43†L129-L137】. Секреты (ключи API, токены) удобно хранить именно в настройках Hugging Face Space.  

# Интеграции n8n: Telegram, Supabase, OpenAI  
**Telegram.** n8n имеет встроенные ноды для работы с Telegram【10†L1522-L1525】: **Trigger**-узел («Telegram Trigger») реагирует на входящие сообщения из бота, а **Telegram**-узел может отправлять/редактировать сообщения, файлы и т.д. При настройке триггера можно **ограничивать список чатов или пользователей**: опции *Restrict to Chat IDs* и *Restrict to User IDs* позволяют указать «белый список» ID, для которых будет срабатывать бот【16†L1601-L1605】. То есть в белом списке указывают либо ID чатов (каналы, группы) либо ID пользователей. Это соответствует вопросу: в «белом списке» Telegram Trigger указываются ID аккаунтов (чатов/пользователей), а не пароли или ключи.  

**Supabase.** n8n поддерживает Supabase «из коробки»【6†L1523-L1531】. Есть базовая **Supabase Node** для CRUD-операций со строками в базе и **Supabase Vector Store Node** для RAG (векторного поиска)【5†L1569-L1577】. Последняя полезна, если нужно хранить эмбеддинги: она позволяет вставлять документы и искать похожие по векторному запросу【5†L1569-L1577】【31†L1567-L1574】. (Можно также использовать SQL-ноду n8n для доступа к Postgres/Supabase напрямую.) Если требуется загрузить файлы/данные в векторное хранилище, n8n предлагает «шаблон RAG» с готовым workflow: в нём сначала «чистится» и разбивается текст, вставляется в Vector Store, а потом при запросе извлекаются наиболее релевантные куски【31†L1556-L1564】.  

**OpenAI (и другие LLM).** n8n имеет ноду **OpenAI** (и **Chat Model**) для общения с моделями GPT【40†L1526-L1533】. Через неё можно посылать запросы (ChatCompletion) и получать ответы. В контексте RAG часто используют *AI Agent* (узел Agent) с подключением инструментов: в таком случае к OpenAI можно «подключить» Vector Store в качестве инструмента – агент будет подмешивать извлечённые из базы фрагменты в ответ【31†L1589-L1596】. Шаблон «OpenAI Agent to Chat With Files In Supabase Storage» на сайте n8n показывает такую связку. Кроме того, есть нода **AI Hub / AI Workflow Builder** для визуальной сборки цепочек, но для простоты можно обойтись связью Trigger → Vector Store → Agent → Telegram.  

# Примерный MVP: архитектура и последовательность действий  
Для минимального жизнеспособного решения логика может быть примерно такой: пользователь пишет боту в Telegram → Telegram Trigger получает сообщение → n8n workflow анализирует запрос (возможно, переводит текст на нужный язык) и ищет нужную информацию в базе. 
- *Запрос к базе:* с помощью **Supabase Vector Store** или SQL-ноды извлекаем релевантные документы по запросу.  
- *Обращение к ИИ:* затем запускаем **OpenAI Agent/Chain** с найденным контекстом. Агенту можно передать фрагменты из базы как «tools» (инструменты) или объединить через Q&A-цепочку.  
- *Формирование ответа:* полученный ответ может быть отформатирован (узел Set или Function) согласно нужному шаблону.  
- *Отправка сообщения:* через Telegram-ноду посылаем ответ пользователю.  

Основные шаги можно оформить списком: 
1. **Webhook/Trigger:** ожидаем входящее сообщение от Telegram.  
2. **Подготовка данных:** из текста создаём эмбеддинг и ищем похожие записи в Supabase.  
3. **Запрос к LLM:** вызываем OpenAI (или Claude) с найденными контекстными данными.  
4. **Вывод результата:** отправляем ответ через ноду Telegram.  

Каждый шаг реализуется «готовыми» узлами n8n, что ускоряет работу (снижаем необходимость писать код с нуля). n8n также позволяет *импортировать* готовые workflow в формате JSON (команда `n8n import:workflow` в CLI【28†L699-L707】), так что теоретически можно генерировать JSON кода для workflow (например, с помощью Codex). На практике проще один раз настроить узлы в UI. Для любой специфики можно добавить **Code**-узел (JavaScript) и написать логику вручную, но это дольше. Python-код из существующего проекта придётся переписать на JS или вызывать по HTTP, т.к. нативной поддержки Python в n8n нет.  

# Диаграмма бизнес-процесса (swimlane)  
Для визуализации процесса рекомендуют «swimlane»-диаграмму, где горизонтальные дорожки (swimlanes) соответствуют ролям или подсистемам【34†L65-L73】. Например, можно сделать дорожки «Пользователь», «Telegram Bot (n8n Trigger)», «n8n (Agent)», «Supabase» и т.д. На диаграмме поэтапно показать, кто и что делает: пользователь пишет запрос, Telegram-бот пересылает его в n8n, n8n достаёт данные из Supabase, запускает ИИ-агента и возвращает ответ через Telegram. Ниже приведены иллюстрации типичных swimlane-диаграмм: они демонстрируют разделение на роли и последовательность действий【35†】【36†】. Такие диаграммы легко строятся в инструментах вроде https://swimlanes.io, где описывают секвенции между «акторами».  

【35†embed_image】 *Рис.1*. Пример swimlane-диаграммы (ролями идут R&D, Marketing, Manufacturing)【34†L65-L73】.

# Ответы на уточняющие вопросы  
- **Краудинтеграции vs свой код.** В n8n лучше по максимуму использовать **готовые узлы**: они быстрее настраиваются через GUI и покрывают базовые задачи (HTTP-запросы, CRM, базы, ИИ и т.д.). Если нужной операции нет, можно использовать узел **HTTP Request** с собственным API или **Function/Code** для JS-логики.  
- **«Шаблоны»**. Если говорилось о шаблонах ответов, то в n8n можно собирать вывод с помощью узла **Set** или встроенных выражений (таких как `{{$json["field"]}}`) – по сути формируя JSON или текст по образцу. Также есть раздел «Templates» для готовых примеров workflow (как в документации OpenAI)【40†L1570-L1578】. Если нужны текстовые шаблоны ответов, можно передать LLM соответствующую инструкцию («answer in format JSON with fields X,Y,Z»).  
- **Белый список Telegram**. Как уже отмечено, в настройках Telegram Trigger ноды можно указать конкретные Chat IDs или User IDs для фильтрации входящих сообщений【16†L1601-L1605】. Это и есть ваш «whitelist» – список ID, от которых будут приниматься сообщения.  
- **Генерация workflow кодом.** Теоретически можно генерировать JSON для n8n workflow с помощью LLM (Codex, Claude), а затем загрузить его через `n8n import`. Однако в условиях жёстких сроков удобнее разметить всё мышью в UI: drag-n-drop узлов и настройка полей. К тому же n8n GUI сразу валидация данных и проверка – быстрее увидеть ошибки.  

Таким образом, MVP можно сделать за несколько дней, используя **готовые узлы n8n** для Telegram, Supabase и OpenAI. **Секреты** (API-ключи, пароли БД) сохраняйте в настройках Spaces или в .env (локально). Бизнес-процесс иллюстрируйте swimlane-диаграммой (см. примеры выше)【34†L65-L73】【16†L1601-L1605】. Это даст ясное представление о том, как взаимодействуют участники системы и что делает n8n на каждом этапе.
