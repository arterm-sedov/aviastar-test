## Что именно делать

В MVP по ТЗ обязательно входят: индексация пакета документов в векторную БД, RAG-пайплайн в n8n с классификацией запроса и ответом **только** при наличии источников и достаточной уверенности, Telegram-бот с авторизацией по белому списку, логирование запросов и базовая аналитика использования.  Отдельно важно соблюсти ограничение: LLM не должна бесконтрольно генерировать финальные юридические документы; для шаблонов нужен кодовый сценарий заполнения полей, а не “свободная генерация договора”.[^1]

- Поднять n8n локально в Docker/WSL для разработки.
- Собрать demo-ready pipeline.

## MVP-границы

Самый безопасный MVP для такого срока — разделить систему на 4 потока.[^1]


| Блок | Что входит в MVP | Что лучше не тащить в MVP |
| :-- | :-- | :-- |
| База знаний | 3–10 demo-документов: локальные регламенты/шаблоны из открытых источников + 1 внешний нормативный источник [^1] | Автообновление базы, сложный ETL, полноценный мониторинг изменений [^1] |
| RAG | Загрузка, chunking 500/50, embeddings, поиск top-k, re-rank/score threshold, ответ с цитатами и отказ при low confidence [^1] | Многоступенчатые агенты, сложная memory/stateful orchestration |
| Документы | 1–2 шаблона: например, “ответ HR по отпуску/увольнению” и “черновик сопроводительного письма/служебной записки” с заполнением полей через код [^1] | Автогенерация договоров “с нуля”, DOCX-конструктор со множеством ветвлений |
| Интерфейс и контроль | Telegram bot, whitelist, env secrets, logging, thumbs up/down, очередь плохих ответов на review [^1] | RBAC, admin panel, Bitrix/Workspace integrations, кабинет сотрудника [^1] |

Практически это означает: **один основной use case HR** и **один основной use case legal**.  Например: “какой порядок оформления отпуска/увольнения/командировки” для HR и “какие документы нужны/какие нормы применимы” для юр./транспортного кейса, потому что ТЗ прямо упоминает трудовое и транспортное законодательство.[^1]

## Как собрать решение

Для тебя как Python-разработчика лучший путь — использовать n8n как оркестратор, а не пытаться всю логику реализовать только нодами. ТЗ допускает выбор реализации, если результат соблюден, поэтому сложные части — нормализацию метаданных, подготовку чанков, возможно even retrieval helper — можно оставить в Python-сервисе, а n8n использовать для ingestion orchestration, Telegram webhook, branching, logging и вызова моделей/API.[^1]

Рекомендуемая архитектура MVP:

- n8n — orchestration layer.
- Supabase pgvector — vector store и, по возможности, таблицы логов/feedback.[^1]
- Python helper service — ingestion/doc parsing/chunking/template fill.
- Telegram bot — пользовательский вход.
- LLM + embedding provider — через API.
- Один внешний источник — либо статический пакет НПА из открытых публикаций, либо заранее выгруженные открытые статьи/разъяснения по HR/legal тематике для демо.[^1]

Минимальный набор workflow в n8n:

1. `ingest_documents`
2. `answer_question`
3. `generate_draft`
4. `save_feedback`
5. `review_bad_answers`

Это хорошо ложится на n8n: ingestion можно запускать вручную или cron, а пользовательские сценарии вести через Telegram trigger/webhook. Для первого этапа локальная self-hosted установка в Docker/WSL выглядит правильнее, потому что отладка credentials, volumes и custom code там проще, чем сразу в Hugging Face Space.

## Логика пайплайна

Основной пользовательский workflow стоит строить так:

1. Telegram message received.
2. Проверка user_id по whitelist.[^1]
3. Классификация запроса: HR / legal / draft request / out of scope.[^1]
4. Если запрос на черновик — переход в template flow.
5. Если вопрос на поиск нормы — retrieval flow.
6. Поиск релевантных чанков в Supabase pgvector с метаданными `file`, `section`, `date`.[^1]
7. Проверка confidence threshold.
8. Если confidence низкий или источников мало — безопасный отказ.
9. Если confidence достаточный — ответ с источниками.
10. Логирование в Supabase или Google Sheets.
11. Добавление inline feedback buttons.
12. Negative feedback → очередь review.[^1]

Ключевая мысль для интервью: ты не делаешь “LLM-юриста”, ты делаешь **guardrailed retrieval assistant**. Это прямо соответствует ТЗ про отказ без источника и запрет на бесконтрольную генерацию финальных документов.[^1]

## Swimlane BPMN

Для BPMN я бы использовал 5 дорожек:

- Сотрудник
- Telegram-бот
- n8n
- База знаний / Supabase
- Юрист/HR-эксперт

В упрощенном виде процесс такой:


| Lane | Шаги |
| :-- | :-- |
| Сотрудник | Отправляет вопрос или запрос на черновик |
| Telegram-бот | Принимает сообщение, передает в n8n, показывает ответ и кнопки оценки |
| n8n | Проверяет whitelist, классифицирует запрос, запускает retrieval или template path, оценивает confidence, формирует ответ/отказ, пишет лог |
| Supabase / KB | Хранит чанки, метаданные, логи, feedback, bad-answer queue [^1] |
| Юрист/HR-эксперт | Разбирает плохие ответы, обновляет документы/шаблоны, подтверждает корректность черновиков |

Текстовая BPMN-структура для схемы:

- Start event: “Поступил запрос от сотрудника”.
- Gateway: “Пользователь авторизован?”
- Если нет → “Отказ в доступе”.
- Если да → Task: “Классифицировать запрос”.
- Gateway: “Тип запроса?”
- Ветка A: “Нормативный вопрос” → retrieval → confidence gateway → ответ с источниками / отказ и эскалация.
- Ветка B: “Запрос черновика” → извлечение шаблона → сбор полей → кодовое заполнение → черновик на проверку.
- После ответа → логирование → feedback.
- Negative feedback → queue for review.
- End event: “Ответ доставлен / отправлен на review”.[^1]

Если делать совсем правильно для показа, добавь еще отдельный подпроцесс “Обновление базы знаний”: загрузка документов → chunking 500/50 → embeddings → запись в pgvector с метаданными.[^1]

## План на 4–5 дней

Реалистичный план такой:

- День 1: локально поднять n8n + Supabase stack, выбрать demo-документы, определить схему metadata.
- День 2: ingestion pipeline, chunking 500/50, загрузка в vector store.[^1]
- День 3: Telegram bot, whitelist, основной Q\&A flow, citations, confidence gate.[^1]
- День 4: draft generation по 1–2 шаблонам, logging, feedback, bad-answer queue.[^1]
- День 5: BPMN swimlane, polish, test scenarios, деплой demo-версии.[^1]

Для собеседования полезно сказать, что Hugging Face Spaces — это скорее демонстрационный контур, а не production-hosting для внутренних юридических/HR данных. Это следует уже из самого характера задачи: есть ограничения доступа, whitelist и чувствительные данные, значит production-контур лучше держать в контролируемой инфраструктуре, а Spaces использовать как витрину с обезличенными demo-данными.[^1]

## Риски

ТЗ отдельно просит описать риски, и здесь лучше перечислить их явно.[^1]

- Качество источников: если demo-документы слабо структурированы или устарели, retrieval будет шумным.[^1]
- Правовой риск: даже при RAG нельзя выдавать ответы как “юридически окончательные”, особенно без human review для документов.[^1]
- Доступ к внешним правовым системам: интеграция с КонсультантПлюс может быть нетривиальной и не нужна для MVP; в ТЗ это можно отнести к следующему этапу.[^1]
- Безопасность: Telegram как интерфейс удобен, но требует строгого whitelist и исключения секретов из кода.[^1]
- Срок: 4–5 дней мало для production-grade решения, поэтому нужна четкая отсечка scope до MVP.[^1]
- Observability: без нормального логирования и review queue трудно улучшать качество, поэтому этот блок нельзя откладывать.[^1]
