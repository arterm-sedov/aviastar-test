AI ассистент для юридического / HR отдела с RAG-
архитектурой (MVP) 
MVP n8n RAG AI-ассистент 
Контекст: компания ведет активную цифровую трансформацию. Юридический и 
HR-отделы работают с нормативной базой, внутренними регламентами, трудовым 
и транспортным законодательством, шаблонами документов. Сейчас это требует 
ручного поиска, базы знаний нет, что создает узкое место: время на подготовку 
документов, риск использования устаревших норм, зависимость от конкретных 
сотрудников-носителей знаний. 
Цель: снизить время подготовки типовых юридических документов и ответов на 
нормативные вопросы за счет AI ассистента, работающего поверх актуальной базы 
знаний компании и внешних нормативных источников. 
ЧТО СДЕЛАТЬ 
/Нарисовать схему БП в swim lanes./ 
 
1.Развернуть векторную БД и проиндексировать пакет документов. 
• Индексация внутренних документов: регламенты, шаблоны договоров, 
приказы, политики.  
• Для демонстрации, документы взять из открытых источников, с 
возможностью быстро обновить и актуализировать информацию реальными 
данными. 
• Подключение внешних источников: КонсультантПлюс, нормативные акты по 
грузовым перевозкам, трудовое законодательство и тому подобное.  
• Все документы векторизуются и хранятся в Supabase (pgvector).  
• Для демо - пакет открытых документов + 1 внешний источник на свой 
выбор в юридическом/hr сегменте. 
• Supabase pgvector. Чанки по 500 токенов. Мета: файл, раздел, дата. 
• TokenTextSplitter(chunk_size=500, chunk_overlap=50) 
 
Supabase/Pinecone - выбор 
 
2. Собрать RAG-pipeline в n8n: вопрос → поиск → ответ с источниками 
Классифицировать тип запроса.  
Ответ без источника или с недостаточным уровнем уверенности (скоринг) не 
выдавать. 
 
Не генерируем финальные юридические документы через LLM без проверки. 1 
галлюцинация из 1000 в договоре может дорого стоить для компании. LLM для 
поиска и ответов, шаблоны - через код с заполнением полей. 
 
3. Сделать Telegram-бот как интерфейс с авторизацией, для исключения 
несанкционированного доступа к данным. 
Белый список.  
API-ключи обязательно в env, не в коде и не в нотах. 
 
4. Логировать все запросы и добавить кнопки оценки ответа 
Google Sheets или Supabase.  
Плохие ответы - в очередь на ревью.  
Запросы - в аналитику для формирования базы знаний и частых кейсов.
